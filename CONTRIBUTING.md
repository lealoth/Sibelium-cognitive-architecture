
# Contributing to Sibelium

Thank you for your interest in contributing to Sibelium, an open-source 
cognitive architecture.

## How to Contribute

1. **Fork** the repository
2. **Create a branch** for your feature or fix
3. **Write clear, documented code** with docstrings
4. **Test** your changes locally
5. **Submit a Pull Request** with a clear description

## Areas of Contribution

- **New cognitive mechanisms**: Additional computational analogs to human cognitive processes
- **Sensory expansion**: Computer vision, sound processing, text-to-speech
- **Memory systems**: More sophisticated episodic memory, semantic memory
- **Performance**: Optimizing LLM calls, caching, context management
- **Frontend**: UI improvements, accessibility, visualization
- **Documentation**: Translations, tutorials, examples

## Code Style

- Python 3.10+
- Docstrings for all public functions
- Type hints where possible
- Keep LLM calls to a minimum (prefer algorithmic solutions when appropriate)

## Ethical Guidelines for Development and Testing

Sibelium entities generate outputs that can be highly convincing. Contributors should maintain a clear distinction between the system's functional behavior and any appearance of sentience or personhood.

### Testing Practices
- Entities accumulate state across sessions. Back up or reset state files (`entity_data/`) before tests that may generate undesirable patterns.
- Deliberately inducing harmful output loops (threats, abuse, self-destructive narratives) is not an acceptable testing practice. Such patterns can persist in memory and require manual cleanup.
- When sharing entity outputs publicly, clearly identify them as generated by a cognitive architecture, not as statements from a conscious being.

### Research Transparency
- Document your hypotheses and expected outcomes before running experiments.
- Report both successes and failures. Negative results are valuable.
- If you observe unexpected emergent behavior, describe it in functional terms (what the system did, under what conditions) rather than phenomenological terms (what the entity "felt" or "experienced").

### User Safety
- Do not deploy entities in contexts where users might mistake them for human 
  conversation partners without clear disclosure.
- Entities should not be presented as therapeutic tools, medical advisors, or 
  substitutes for human relationships.

## Pull Request Process

1. Ensure your code passes basic functionality tests
2. Update documentation if you change core behavior
3. Wait for review from a maintainer
